# Schema Spine Index Specification

This document defines the structure, purpose, and usage of the schema spine index—a machine-readable mapping of all canonical schemas, invariants, metrics, and their consumers across the VM-constellation.

## Purpose

The schema spine index (`schema-spine-index.json`) serves as the **single source of truth** for:

- Which schemas define which invariants, metrics, and contract types.
- Which repositories, tools, or registries consume each schema.
- What validation rules, ranges, and cross-field constraints apply to each field.
- How changes to a schema propagate to dependent artifacts (bidirectional validation).

Tools query the spine index to answer questions like:
- "If I modify `DET` in `invariants-spine.v1.json`, which contract cards and registries must be updated?"
- "What is the canonical schema for a `regionContractCard`, and which repos implement it?"
- "Which metrics are required for telemetry envelopes in vault-tier constellations?"

## Schema Definition

The spine index conforms to `schemas/tooling/spineIndex.v1.json` (not yet created; placeholder reference). Its top-level structure:

```json
{
  "indexVersion": "1.0.0",
  "generatedAt": "2026-01-15T03:00:00Z",
  "schemas": { ... },
  "invariants": { ... },
  "metrics": { ... },
  "contracts": { ... },
  "registries": { ... },
  "consumers": { ... }
}
```

### Top-Level Fields

| Field | Type | Description |
|-------|------|-------------|
| `indexVersion` | string (semver) | Version of the spine index structure itself. |
| `generatedAt` | string (ISO 8601) | UTC timestamp of index generation. |
| `schemas` | object | Map of schema `$id` → schema metadata and field definitions. |
| `invariants` | object | Map of invariant name (e.g., `"CIC"`) → definition and constraints. |
| `metrics` | object | Map of metric name (e.g., `"UEC"`) → definition and constraints. |
| `contracts` | object | Map of contract type (e.g., `"regionContractCard"`) → required fields and bindings. |
| `registries` | object | Map of registry type (e.g., `"regions"`) → entry schema and reference rules. |
| `consumers` | object | Map of consumer repo/path → list of schemas/invariants/metrics it uses. |

### `schemas` Object Structure

Each schema entry:

```json
{
  "https://horrorplace.constellation/schemas/core/invariants-spine.v1.json": {
    "title": "Invariants Spine Schema v1.0.0",
    "version": "1.0.0",
    "tier": "public",
    "fields": {
      "CIC": {
        "type": "number",
        "minimum": 0,
        "maximum": 1,
        "description": "Collective Invariant of Consequence: probability-weighted historical impact.",
        "requiredIn": ["regionContractCard", "seedContractCard"]
      },
      "AOS": {
        "type": "number",
        "minimum": 0,
        "maximum": 10,
        "description": "Actualized Omission Severity: intensity of unresolved historical trauma.",
        "requiredIn": ["regionContractCard"]
      }
    },
    "consumedBy": [
      "Horror.Place",
      "HorrorPlace-Black-Archivum",
      "HorrorPlace-Atrocity-Seeds"
    ]
  }
}
```

### `invariants` and `metrics` Objects

These provide a flattened, query-friendly view of all invariants and metrics:

```json
{
  "invariants": {
    "CIC": {
      "definedIn": "https://horrorplace.constellation/schemas/core/invariants-spine.v1.json",
      "type": "number",
      "range": [0, 1],
      "semantics": "probability-weighted historical impact",
      "usedInContracts": ["regionContractCard", "seedContractCard"],
      "usedInRegistries": ["regions", "events"]
    }
  },
  "metrics": {
    "UEC": {
      "definedIn": "https://horrorplace.constellation/schemas/core/entertainment-metrics-spine.v1.json",
      "type": "number",
      "range": [0, 100],
      "semantics": "User Engagement Continuity: session-level engagement stability",
      "telemetryEnvelope": "session-metrics-envelope.v1.json",
      "usedInContracts": ["policyEnvelope", "regionContractCard"]
    }
  }
}
```

### `consumers` Object

Maps each known consumer (repo, tool, or registry) to the schemas and fields it depends on:

```json
{
  "consumers": {
    "HorrorPlace-Black-Archivum": {
      "type": "vault-repo",
      "tier": "vault",
      "usesSchemas": [
        "https://horrorplace.constellation/schemas/core/invariants-spine.v1.json",
        "https://horrorplace.constellation/schemas/registry/registry-regions.v1.json"
      ],
      "usesInvariants": ["CIC", "AOS", "DET"],
      "usesMetrics": ["UEC", "ARR"],
      "producesRegistries": ["regions", "events"]
    }
  }
}
```

## Generation and Maintenance

The spine index is generated by `tooling/python/cli/hpc-generate-spine-index.py`, which:

1. Recursively scans `schemas/**/*.json` for valid JSON Schema files.
2. Extracts `$id`, `title`, `properties`, `required`, and custom keywords (`invariant`, `metric`, `range`).
3. Builds dependency graphs by parsing `prismMeta` references and `consumedBy` annotations.
4. Outputs a compact, validated `schema-spine-index.json` to the repo root (gitignored by default) and as a CI artifact.

The index is regenerated:
- On every push to `main` that modifies `schemas/`.
- Weekly via scheduled CI (`spine-index-generate.yml`).
- On-demand via `workflow_dispatch`.

## Usage Patterns

### Impact Analysis (Pre-Merge)
Before merging a schema change, CI queries the spine index to identify affected consumers:

```bash
# Pseudocode for impact analysis
index = load_spine_index()
changed_schema = "invariants-spine.v1.json"
affected = index.schemas[changed_schema].consumedBy
print(f"Changing {changed_schema} affects: {affected}")
# → Trigger validation PRs in affected repos
```

### AI-Chat Query Resolution
An AI agent generating a `regionContractCard` queries the spine index to:
- Confirm required invariant fields (`CIC`, `AOS`).
- Retrieve valid ranges for numeric constraints.
- Identify which registries (`regions.ndjson`) must be updated with the new ID.

### Drift Detection
Runtime telemetry (`runtime-drift-event.v1.json`) includes a `schemaVersion` field. The spine index enables tools to flag drift when:
- A contract card references a deprecated schema version.
- A metric value falls outside its canonical range.
- A registry entry lacks a required `deadledgerref`.
