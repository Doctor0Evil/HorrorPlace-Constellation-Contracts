# CHAT_DIRECTOR Specification v1

This document describes the v1 behavioral and data model specification for CHAT_DIRECTOR. It is intended as a reference for implementers, AI tooling authors, and maintainers of the Constellation-Contracts stack.

---

## Data model

### Object kinds

CHAT_DIRECTOR works with a set of contract and manifest object kinds, defined via JSON Schema and referenced by `$id` or `schemaRef`. Common examples:

- `regionContractCard`
- `seedContractCard`
- `moodContract`
- `eventContractCard`
- `personaContractCard`
- Associated manifest types (e.g., registry manifests, routing manifests)

Each object kind:

- Has a v1 schema under `schemas/core/`.
- Declares optional annotations such as `x-objectKind`, `x-contractFamily`, and `x-tierApplicability`.
- Is linked to invariants and metrics via the schema spine.

### Invariants

Invariants quantify spectral and historical properties of locations and contracts, including:

- CIC – Catastrophic Imprint Coefficient
- MDI – Mythic Density Index
- AOS – Archival Opacity Score
- RRM – Ritual Residue Map strength
- FCF – Folkloric Convergence Factor
- SPR – Spectral Plausibility Rating
- RWF – Reliability Weighting Factor
- DET – Dread Exposure Threshold
- HVF – Haunt Vector Field magnitude/direction
- LSG – Liminal Stress Gradient
- SHCI – Spectral-History Coupling Index

Their canonical ranges by contract family and tier are defined in `invariants-spine.v1.json` and folded into `schema-spine-index-v1.json`.

### Entertainment metrics

Entertainment metrics describe player-facing pacing and cognitive load:

- UEC – Uncertainty Engagement Coefficient
- EMD – Evidential Mystery Density
- STCI – Safe-Threat Contrast Index
- CDL – Cognitive Dissonance Load
- ARR – Ambiguous Resolution Ratio

Ranges and recommended bands per contract family and tier are defined in `entertainment-metrics-spine.v1.json` and exposed via the spine index.

## Spine index contract

`schema-spine-index-v1.json` is the canonical link between schemas, invariants, and metrics. It is generated by `spineindexbuilder.py` and must contain:

- A list of schema entries:
  - `schema_id`
  - `file_path`
  - `title`
  - `object_kind`
  - `contract_family`
  - `tier_applicability[]`
  - `invariants[]`
  - `metrics[]`
- `invariants_catalog[]`:
  - `key`
  - `contract_family`
  - `tier`
  - `min`
  - `max`
- `metrics_catalog[]`:
  - `key`
  - `contract_family`
  - `tier`
  - `min`
  - `max`

CHAT_DIRECTOR relies on this file at startup to:

- Answer “which invariants/metrics matter for this kind/tier?”
- Enforce band checks for contract artifacts.
- Guide AI authoring tools towards acceptable numeric ranges.

No other component should attempt to construct this file independently.

## Authoring envelopes

Two primary envelope types define AI authoring interactions:

- `ai-authoring-request`:
  - Required fields: `intent`, `objectKind`, `targetRepo`, `tier`, `schemaRef`, `referencedIds`.
  - Intended invariant and metric bands via `intendedInvariants` and `intendedMetrics`.
  - Optional manifest hints like `manifestTier`.

- `ai-authoring-response`:
  - Exactly one primary artifact (contract card) encoded as `artifact` or `artifacts` with `isPrimary`.
  - Envelope metadata: provenance, agent, session, `targetRepo`, `path`, `tier`, etc.
  - Artifact fields: `objectKind`, `schemaRef`, `invariantBindings`, `metricTargets`, and any registry references.

The Python validator (`aiauthoringvalidator.py`) enforces envelope structure, presence of required fields, and band checks according to the spine.

## CLI behavioral spec

At a high level, the `hpc-chat-director` CLI is expected to expose:

- `plan`:
  - Input: `ai-authoring-request` envelope.
  - Output: normalized plan with resolved `targetRepo`, `targetPath`, manifests, and band guidance.

- `validate-response`:
  - Input: `ai-authoring-response` envelope.
  - Output: diagnostics for schema, invariant, metric, manifest, and envelope issues.

- `apply`:
  - Input: validated response or plan+artifact.
  - Effect: writes changes into registries according to manifests, respecting trust and phase rules.

The CLI should support structured JSON output with error codes, JSON Pointers, and hints suitable for AI-driven remediation.

## Validation tiers

Validation is intentionally layered:

1. **Schema-only**:
   - `hpc-validate-schema.py`
   - Ensures schemas are well-formed and references resolve.

2. **Registry structure and references**:
   - `hpc-lint-registry.py`
   - Ensures NDJSON entries match registry schemas and reference existing IDs.

3. **Envelope pre-flight**:
   - `aiauthoringvalidator.py`
   - Quick checks on AI envelopes and basic bands.

4. **Full CHAT_DIRECTOR validation**:
   - `hpc-chat-director validate-response`
   - Manifest, invariant, and routing enforcement.

This layering supports fast feedback loops for AI chats while preserving strong guarantees at the Rust layer.
