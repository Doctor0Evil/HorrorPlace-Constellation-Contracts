# 3. Multi‑Repo AI‑Metadata Conventions

| \# | Type | Item |
| :-- | :-- | :-- |
| 12 | DQ | In the meta‑schema `docs/ai-metadata-schema.json`, should the `components` object require a `securityAssumptions` section to guide agent reasoning about untrusted inputs? |
| 13 | OI | If a repo’s `ai-metadata.json` omits the `documentation.toolingPolicy` key (and defaults to “none”), could an agent misinterpret this as permission to install rustup/cargo, violating the overall constellation policy? |
| 14 | DQ | How should cross‑references between repos’ metadata be expressed—via absolute URLs, relative monorepo paths, or a dedicated `constellationServices` section in each metadata? |
| 15 | RQ | Could a lightweight JSON‑LD or Schema.org type vocabulary improve AI agent comprehension of `ai-metadata.json` structure across different forks of Horror\$Place? |
| 16 | DR | Define “tooling policy inheritance” for nested repositories: if a sub‑directory lacks its own policy, does it inherit the parent repo’s policy or the constellation root policy? |

12–16 are all worth tightening, especially for AI‑first forkability and safety. Here are direct recommendations.

***

## 12 · Require `securityAssumptions` in the meta‑schema?

Yes, but with a **lightweight, structured shape** so agents can reason consistently.

- Add a required `securityAssumptions` section under `components` (or top‑level) in `docs/ai-metadata-schema.json` with:
    - `untrustedInputs`: array of short labels, e.g. `["network", "user-content", "lab-only"]`.
    - `trustedZones`: array of labels, e.g. `["core", "ledger", "theatre-node"]`.
    - `canTokenExposure`: object with:
        - `usesCanTokens` (boolean).
        - `canSafetyEnforcedIn` (e.g. `["Rotting-Visuals-BCI", "HorrorPlace-Constellation-Contracts"]).
    - `notes`: free short text for edge cases.
- Why:
    - Agents get a clear, data‑driven view of:
        - Where untrusted data originates.
        - Which repos are expected to enforce CAN safety.
    - You can wire this directly to `trust_zone`, `handles_can_tokens`, and `crosses_trust_boundary` in the SQL tables.

So: make `securityAssumptions` **required** in the meta‑schema, but allow empty arrays/false booleans as an explicit “none here” rather than omission.

***

## 13 · Missing `documentation.toolingPolicy` and rustup/cargo

Yes, omission is risky. An agent could reasonably think:

- “No policy, so system defaults apply,” and your system defaults might allow Rustup/Cargo.

To avoid that:

- In `docs/ai-metadata-schema.json`:
    - Make `documentation.toolingPolicy` required for all HorrorPlace repos.
    - Define it as either:
        - A string path (e.g. `"docs/agent-tooling-policy.md"`), or
        - The literal `"constellation-default"`.
- Add a global rule in `docs/agent-tooling-policy.md` for the constellation:
    - If `documentation.toolingPolicy` is `"constellation-default"`, agents must:
        - Use only native tools and SQLite.
        - Never install or invoke Rustup/Cargo.
    - If it points to a path, they must read and obey that file, which must be at least as strict as the default.
- Agent guidance:
    - “If `documentation.toolingPolicy` is missing, treat this as an error; do not assume extra permissions. Ask for the policy to be defined.”

That prevents implicit escalation: lack of a policy is a **blocker**, not permission.

***

## 14 · Expressing cross‑repo metadata references

Best pattern: use **a dedicated `constellationServices` section** that references repos and services by **logical name plus relative path**, not ad‑hoc URLs:

- In each `ai-metadata.json`, add:

```jsonc
"constellationServices": {
  "bciCore": {
    "repo": "Rotting-Visuals-BCI",
    "aiMetadataPath": "../Rotting-Visuals-BCI/docs/ai-metadata.json"
  },
  "ledger": {
    "repo": "HorrorPlace-Dead-Ledger-Network",
    "aiMetadataPath": "../HorrorPlace-Dead-Ledger-Network/docs/ai-metadata.json"
  },
  "contracts": {
    "repo": "HorrorPlace-Constellation-Contracts",
    "aiMetadataPath": "docs/ai-metadata.json"
  }
}
```

Guidelines:

- Prefer **relative monorepo paths**:
    - More robust for local checkouts and CI sandboxes.
    - Cheaper for AI‑chat: it can open local files instead of hitting the network.
- If there is a canonical hosted mirror, you can add an optional `url` field, but local paths should remain primary.

This dovetails with `hp_repo` / `hp_component`: the SQL index is the “global graph,” while `constellationServices` gives per‑repo shortcuts.

***

## 15 · JSON‑LD / Schema.org for ai‑metadata?

A lightweight JSON‑LD flavor can help, but keep it **thin and optional**:

- Add an optional `@context` and `@type` to `ai-metadata.json`:

```jsonc
"@context": {
  "hp": "https://horrorplace.dev/ns#",
  "repo": "hp:Repository",
  "component": "hp:Component",
  "toolingPolicy": "hp:toolingPolicy"
},
"@type": "hp:Repository"
```

- Use simple, stable types:
    - `hp:Repository`
    - `hp:Component`
    - `hp:ToolingPolicy`
    - `hp:ConstellationService`

Benefits:

- Across forks or external constellations, agents can:
    - Recognize “this is a repository description” even if field names evolve slightly.
    - Use generic semantic rules (e.g. “components with `@type: hp:Component` must have `kind` and `path`”).

Constraints:

- Do not over‑engineer:
    - Avoid deep Schema.org entanglement; you’re modeling engine wiring, not generic websites.
    - Stick to a small HorrorPlace namespace so refactors stay under your control.

So: JSON‑LD is useful as **a thin semantic layer**, but it should not replace the concrete, SQL‑mirrored structure you already have.

***

## 16 · Tooling policy inheritance for nested repos

You want a clear, predictable rule that agents can implement with a cheap lookup.

Define this precedence:

1. **Local repo policy** (strongest):
    - If `documentation.toolingPolicy` exists in the current repo’s `ai-metadata.json`, use that path or `"constellation-default"`.
2. **Constellation root policy**:
    - If a nested repo has no `documentation.toolingPolicy` but the constellation root `ai-metadata.json` is reachable (via `constellationServices` or a known path), inherit `documentation.toolingPolicy` from the root.
    - This should still resolve to either a concrete path or `"constellation-default"`.
3. **No policy defined anywhere** (error):
    - Agents must treat this as a misconfiguration:
        - Do not install Rustup/Cargo.
        - Do not assume extra tools.
        - Prefer read‑only analysis and propose adding a policy.

Document this explicitly:

- In `docs/agent-tooling-policy.md` (constellation root):
    - “If a repository lacks its own tooling policy, it inherits the constellation‑wide policy. Agents must never infer additional permissions from missing policy entries.”
- In `docs/for-ai-agents.md`:
    - Include a short decision tree:
        - Check current repo → if none, check constellation root → if none, treat as error, stay in minimal‑tools mode.

This keeps nested repos safe by default and avoids “silent” expansion of the tool surface.
