# Telemetry: Resurrection Run v1

This document explains the `resurrection-run.v1` telemetry schema in plain language. It is written for engine teams, tools authors, and AI‑chat agents that need to log, process, or reason about resurrection attempts in HorrorPlace‑aware engines.

The canonical JSON Schema lives at:

- `schemas/telemetry/resurrection-run.v1.json`

All telemetry rows that conform to this schema represent a single resurrection attempt: one engine decision to resurrect or replay a Seed, event, or persona echo, plus the player’s observed response and local quality scoring.

---

## 1. Purpose

Resurrection telemetry serves three main goals:

1. Make repetition and echo logic measurable. Every time the system tries to bring something back (Seed, event, persona), it records why, whether it was allowed, and how it landed.
2. Connect entertainment metrics to concrete attempts. For each resurrection, we capture UEC, EMD, STCI, CDL, and ARR before and after, so we can see whether the attempt moved the experience in the intended direction.
3. Feed resurrection quality back into selection. Aggregated resurrection runs drive `resurrection-aggregate.v1` stats, which in turn influence selector rules and contract tuning.

Resurrection runs are telemetry slices only. They contain no raw horror content or narrative text, only IDs, numeric invariants, metrics, and small categorical labels.

---

## 2. Identity and Context

These fields identify a single resurrection attempt and place it in context.

- `runId`  
  A unique identifier for this resurrection telemetry row. Typically generated by the engine or orchestration layer. Must be unique within a log file or session.

- `sessionId`  
  Identifier for the play session. Used to group multiple resurrection runs from the same continuous play period.

- `playerId`  
  An opaque ID for the player or player slot (e.g., “p1”, “client‑1234”). No PII.

- `seedId`  
  Identifier of the Seed or sequence contract being resurrected or echoed. This usually matches an ID from Atrocity‑Seeds or the core Horror.Place registries.

- `eventId` (optional)  
  Identifier of the specific event contract being resurrected, if applicable.

- `originId`  
  Identifier of the ancestor or original item being echoed. This may be the same as `seedId` or `eventId` or may refer to an older contract version or distinct ancestor.

- `resurrectionKind`  
  Categorical label describing what is being resurrected. Expected values:
  - `seed` – resurrection of a Seed‑level sequence.
  - `event` – resurrection of a specific event contract.
  - `persona` – resurrection of a persona manifestation.
  - `mixed` – composite resurrection involving multiple surfaces.

- `regionId`  
  Region identifier from the region registry. Ties this run to a specific world area and its invariant bundles.

- `tileId`  
  Tile or hex identifier inside the region. This should be stable across builds for the same logical location.

- `stage`  
  The local horror stage in which the resurrection occurs. Aligns with the five‑stage sequence model:
  - `outer`
  - `threshold`
  - `locus`
  - `rupture`
  - `fallout`

- `timePhase`  
  The diegetic time context. Common values:
  - `day`, `dusk`, `night`, `dawn`, `storm`, `other`

These fields let tools answer questions like: “How often was Seed X resurrected in locus/night conditions for this region?”

## 3. Invariants Snapshot

`invariants` captures the local state of the history layer and spatial stress when the resurrection attempt occurs. All numeric values are normalized to the 0.0–1.0 range, consistent with the core invariant schema.

Object: `invariants`

- `CIC` – Catastrophic Imprint Coefficient  
  How strongly the location is imprinted by catastrophic events.

- `MDI` – Mythic Density Index  
  Density of myths, rumors, and legendary narratives attached to this location.

- `AOS` – Archival Opacity Score  
  How opaque the historical record is: missing data, redactions, contradictions.

- `RRM` – Ritual Residue Map strength  
  Intensity of ritual traces and repeated behaviors in this spot.

- `FCF` – Folkloric Convergence Factor  
  How strongly local stories converge on similar motifs and explanations.

- `SPR` – Spectral Plausibility Rating  
  How plausible spectral or anomalous manifestations are here.

- `RWF` – Reliability Weighting Factor  
  How trustworthy the local record and inferred history are.

- `DET` – Dread Exposure Threshold  
  Local cap on safe dread exposure, used to manage cumulative intensity.

- `LSG` – Liminal Stress Gradient  
  Stress felt at thresholds, borders, or transitions around this tile.

- `SHCI` – Spectral‑History Coupling Index  
  How tightly any spectral behavior should be constrained by local history.

- `HVF` – Haunt Vector Field summary  
  Object with:
  - `magnitude` – normalized intensity of directional haunt pressure.
  - `direction` – opaque label describing directional flow (for example: “edge‑to‑locus”, “safe‑to‑unsafe”).

The invariants snapshot is taken at the moment of the resurrection decision. It lets analysis see which invariant profiles produce good or bad resurrection quality.

## 4. Entertainment Metrics Before and After

The schema records entertainment metrics immediately before and after the resurrection attempt.

Objects:

- `metricsBefore`
- `metricsAfter`

Each has:

- `UEC` – Uncertainty Engagement Coefficient  
- `EMD` – Evidential Mystery Density  
- `STCI` – Safe‑Threat Contrast Index  
- `CDL` – Cognitive Dissonance Load  
- `ARR` – Ambiguous Resolution Ratio  

All values are normalized 0.0–1.0. Engines must clamp values into this range.

Typical usage:

- Compute realized deltas as `metricsAfter` minus `metricsBefore`.
- Compare realized deltas to the intended metric targets defined in Seed or event contracts.
- Use these deltas to measure whether resurrected sequences still do useful work (for example, raising UEC without overshooting CDL or collapsing ARR).

Optionally, runtimes may also compute and store a derived `metricsDelta` object, but this is not required by the base schema.

## 5. Resurrection Gate

The `gate` object records how the resurrection gate evaluated this attempt.

Object: `gate`

- `allowed`  
  Boolean. True if the resurrection was permitted to proceed, false if it was blocked.

- `distance`  
  Numeric distance between this candidate and its ancestor or prior manifestations. The exact metric is defined by the resurrection logic (for example, distance over invariants and metrics).

- `noveltyThreshold`  
  The threshold used in this run to decide if the candidate is “novel enough.” A candidate may be blocked if `distance` is below this threshold.

- `reason`  
  Free‑text or short string explaining the gate outcome. Engine teams may standardize values like “too‑similar”, “cooldown‑active”, or “safety‑cap”.

Implementations may extend `gate` with additional fields, such as a structured `reasonCode` and `cooldownSecondsRemaining`, as long as they do not violate the base schema.

---

## 6. Outcome and Physio Summary

The `outcome` object describes how the player responded to the resurrection and how long they were exposed.

Object: `outcome`

- `label`  
  Categorical description of the player’s observed behavior. Common labels:
  - `ignored`
  - `glanced`
  - `observed`
  - `investigated`
  - `fled`
  - `froze`
  - `died`

- `exposureSeconds`  
  Approximate duration, in seconds, for which the player was exposed to the resurrected content (for example, in view or within active audio radius).

- `physioSummary` (optional)  
  Object with summary bands for physiological signals, if available. Example fields:
  - `hrBand` – heart rate band label (“baseline”, “elevated”, “spike”).
  - `edaBand` – electrodermal activity band (“low”, “medium”, “high”).
  - `bciband` – derived BCI state band, aligned with existing BCI contracts.

These fields allow labs and tools to correlate resurrection quality with player behavior and physiology without logging raw sensor streams.

---

## 7. Local Resurrection Quality Score

The `quality` object encodes a local quality assessment for this single resurrection attempt, computed by the runtime logic.

Object: `quality`

- `localScore`  
  Overall quality score in 0.0–1.0. Higher means “better” resurrection by project‑defined criteria.

- `novelty`  
  Score reflecting how novel the resurrected manifestation was, usually derived from `distance` and related filters.

- `metricFit`  
  Score describing how closely realized metric changes matched intended metric targets (for example, from the Seed or event contract).

- `safety`  
  Score describing adherence to safety caps, especially DET and CDL. High safety means the resurrection stayed within configured bounds.

- `ambiguity`  
  Score describing how well the attempt maintained desired ambiguity (ARR) instead of over‑resolving or flattening the mystery.

Implementations should keep the scoring formula documented in a separate spec so different tools and languages can produce compatible scores for the same inputs. The schema does not enforce a particular formula, only the ranges and field presence.

---

## 8. Timestamps

The `timestamps` object anchors the resurrection attempt in time.

Object: `timestamps`

- `recordedAt`  
  ISO‑8601 UTC timestamp when the telemetry row was recorded.

- `sessionTimeSeconds`  
  Optional numeric value representing elapsed time in seconds since the start of the session.

These fields let tools visualize when resurrections happen within a session and correlate them with other telemetry.

## 9. Usage Patterns

### 9.1 Engine Logging

Engines should log one `resurrection-run.v1` object each time they evaluate a resurrection gate, regardless of whether the attempt is allowed or blocked.

Typical flow:

1. Gather identity and context (`sessionId`, `playerId`, `seedId`, `regionId`, `tileId`, `stage`, `timePhase`).
2. Sample local invariants and current metrics.
3. Run resurrection gate logic to determine `allowed`, `distance`, and `noveltyThreshold`.
4. If allowed, execute the resurrection and observe `outcome` and updated metrics.
5. Compute `quality` scores based on novelty, metric fit, safety, and ambiguity.
6. Emit a JSON object conforming to `resurrection-run.v1.json` to an NDJSON log.

### 9.2 Aggregation and Selection

Offline tools (for example, `telemetry_resurrection_aggregate.lua`) read NDJSON logs of resurrection runs, group by `(seedId, resurrectionKind)`, and compute aggregates such as:

- Attempt counts and allow/block rates.
- Average and percentile `localScore`.
- Average metric deltas and safety overflow counts.

These aggregates are written as `resurrection-aggregate.v1` rows and used by selectors and authoring tools to:

- Favor high‑quality resurrections in future sessions.
- Identify Seeds and events whose resurrections consistently overshoot safety caps or fail to move metrics.
- Drive contract tuning via CHATDIRECTOR and AI‑assisted authoring.

---

## 10. Guidance for AI‑Chat Tools

When an AI‑chat agent is asked to work with resurrection telemetry, it should:

- Treat `resurrection-run.v1.json` as the authoritative shape for single attempts.
- Never introduce new top‑level fields without first extending the schema.
- Use the documented invariants and metrics fields instead of inventing new names.
- Reference this document when explaining or generating logging code, so engine teams and CI can validate outputs.

When analyzing logs, AI‑chat tools should focus on:

- Patterns in `localScore` and overflow counts per `seedId` and `resurrectionKind`.
- Misalignment between `metricsBefore`/`metricsAfter` and contract targets.
- Suggestions for contract adjustments (for example, changing metric targets or gating thresholds) rather than rewriting code first.

This keeps resurrection telemetry aligned with the broader HorrorPlace doctrine: invariants and metrics first, contracts before code, and telemetry‑driven refinement of horror behavior.
